==== [Issue Title] ====
proposal: cmd/vet warns when the cutset of strings.{Trim,TrimLeft,TrimRight} is not in order

==== [Issue Body] ====
The proposal is to extend  https://github.com/golang/go/issues/46533 with a warning when the runes in a cutset argument of strings.{Trim,TrimLeft,TrimRight} are not in a canonical sorted order.

https://github.com/golang/go/issues/46533 warns on duplicate runes. This is not sufficient to catch bugs such as https://github.com/ardanlabs/graphql/pull/2/files :
```
strings.TrimRight(url, "/graphql") -> strings.TrimSuffix(url, "/graphql")
```
The cutset "/graphql" does not have duplicate runes. We could warn this example if cmd/vet warned/suggested whenever the cutset does not seem to be treated like a set. A straightforward approach to encourage users to think of these as sets is to have the runes be sorted, e.g. "/aghlpqr".

The huge disadvantage of this approach, and why it is a proposal, is it has a good chance of warning on > half of the currently correct usages of `strings.Trim` with more than 1 rune [by chance]. This is a lot of noise and is a big disadvantage.

Another possible disadvantage is that the sorted order is less readable. Not clear if this is a realistic concern.

==== [Comments] ====

--- Comment #1 by ianlancetaylor ---
Duplicate runes are a clear error.  Unsorted runes are not.  This might be suitable for some static checker tools, but for vet we normally try to stick to only reporting cases that are clearly errors.

That said, it would be interesting to know how many false positives showed up in some corpus.

--- Comment #2 by timothy-king ---
@ianlancetaylor That is a very good point. Unsorted runes are not a clear error. There is nothing wrong with "\t\n" vs "\n\t".

There is something a bit dissatisfying that a human can eyeball `strings.TrimRight(url, "/graphql")` and conclude it is almost certainly a bug. And it would be nice to steer folks to a correct solution. Suggesting sorting would do this, but it might be too stylistic to meet the criteria for a vet checker. Maybe there is a less restrictive heuristic that almost always indicates an error?

--- Comment #3 by rsc ---
Vet cannot catch every last bug. This is one that I think vet simply has to admit defeat on.
